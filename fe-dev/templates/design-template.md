## 五、关键决策【强烈建议】

<!--
- 在多个方案中选择的决策点、权衡、MVP 取舍、容易踩的坑
- 每条决策必须带"决定 + 理由"；有替代方案时列出
- 只有"描述是什么"没有"为什么"的内容 → 不属于本节，放到三、四章
-->

### 5.1 选型与权衡

<!--
示例：
**选 hash-wasm 不选 spark-md5**
- 决定：MD5 计算使用 hash-wasm
- 理由：WASM 实现速度为 spark-md5 的 3-5 倍，大文件场景明显；按需引入 ≈6KB JS
- 替代方案：spark-md5（放弃，性能瓶颈）
-->

### 5.2 MVP 取舍

<!--
- 本次做什么、不做什么、为什么
- 示例：
**OSS PUT 失败 MVP 不做自动重试**
- 理由：失败原因复杂（网络抖动/CORS/签名过期），盲重试可能放大问题；先看线上失败率再决定
- 替代：由调用方暴露"重试"按钮，用户主动触发
-->

### 5.3 经验与坑点【可选】

<!--
- 实现中会踩但容易忽略的点。属于"隐性知识显性化"
- 示例：
**axios 对非 FormData 的请求体会自动 JSON 序列化**
- 现象：会把 File 变成 {}，上传 0 字节
- 解决：必须用独立 axios 实例 + `transformRequest: [data => data]` 原样透传
-->
